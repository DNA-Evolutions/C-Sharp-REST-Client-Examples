# Generating the API Knowledge Base File

This file describes how to regenerate `net_model_knowledge.txt`, a single text file containing all generated OpenAPI model and API classes. It is used as a knowledge base for AI tools.

## What it contains

- All C# files from `src/Org.OpenAPITools/Api/` (REST API client classes: HealthApi, JobApi, OptimizationApi, StreamApi)
- All C# files from `src/Org.OpenAPITools/Model/` (data transfer objects: Node, Resource, RestOptimization, Solution, etc.)
- Each file is separated by a banner showing the source file path
- API and Model sections are grouped under separate headers

## How to regenerate

Run the following command from the repository root:

```bash
output="net_model_knowledge.txt" && echo "" > "$output" && echo "# JOpt TourOptimizer - C# REST Client Knowledge Base" >> "$output" && echo "# Generated from OpenAPI spec version 1.3.5-SNAPSHOT" >> "$output" && echo "# This file contains all generated API and Model classes for AI knowledge ingestion." >> "$output" && echo "" >> "$output" && echo "###############################################################################" >> "$output" && echo "# API CLASSES" >> "$output" && echo "###############################################################################" >> "$output" && echo "" >> "$output" && for f in $(find src/Org.OpenAPITools/Api -name "*.cs" | sort); do echo "===============================================================================" >> "$output" && echo "FILE: $f" >> "$output" && echo "===============================================================================" >> "$output" && cat "$f" >> "$output" && echo "" >> "$output" && echo "" >> "$output"; done && echo "###############################################################################" >> "$output" && echo "# MODEL CLASSES" >> "$output" && echo "###############################################################################" >> "$output" && echo "" >> "$output" && for f in $(find src/Org.OpenAPITools/Model -name "*.cs" | sort); do echo "===============================================================================" >> "$output" && echo "FILE: $f" >> "$output" && echo "===============================================================================" >> "$output" && cat "$f" >> "$output" && echo "" >> "$output" && echo "" >> "$output"; done
```

## When to regenerate

Regenerate after updating the OpenAPI spec and re-running the code generator. Update the spec version comment in the command if needed.


# How to Condense API Knowledge Base Files

This document describes the condensing strategy used for JOpt.TourOptimizer generated client files across Java, TypeScript/Angular, Python, and C#/.NET. Follow these rules to produce a condensed file suitable for AI training while retaining all semantically meaningful content.

---

## General principles

**Remove once, not per file.** Any block that is identical across all files in the knowledge base (license headers, API description docstrings, standard imports) should be removed from every file and, if informative, included exactly once in the file header.

**Keep all semantic content.** Field names, field descriptions, class summaries, method names, method descriptions, parameter names, return types, and discriminator values are all semantically meaningful and must be kept.

**Keep structural content.** Class declarations, inheritance, interface implementations, enum values, and constructor signatures reveal the model's shape and must be kept.

**Remove mechanical boilerplate.** Code that exists purely to satisfy a code generator pattern — serialization helpers, equals/hashCode, HTTP plumbing, import lists — adds no training value and must be removed.

---

## File header

Every condensed output must begin with a header block containing:

```
# JOpt.TourOptimizer — <Language> Client Knowledge Base

Generated from: <source path> (<N> files)
API version: <version>
Generator: <generator name and options>

Common omissions: <comma-separated list of what was removed per file>

---
```

The common omissions line prevents confusion about why certain constructs are absent.

# How to condense the resulting file

## What to remove

### Always remove (identical in every file)
- The large license/API-description comment block at the top of every generated file. This is the multi-kilobyte block describing endpoints, deployment modes, tenant isolation, and encryption — it is identical in every file and belongs in the header once.
- The `# coding: utf-8` / `// Generated by` header lines.
- All standard library import statements (`import`, `using`, `from X import Y`) that are mechanical and language-standard.

### Always remove (pure boilerplate, zero semantic value)
| Language | Remove |
|---|---|
| **Java** | `JSON_PROPERTY_*` constants, `@JsonProperty`/`@JsonInclude` annotations, `@JsonPropertyOrder`, `@jakarta.annotation.Generated`, fluent builder setters, `equals()`, `hashCode()`, `toString()`, `toIndentedString()` |
| **TypeScript** | `addToHttpParams`/`addToHttpParamsRecursive` helpers, three overload signatures per method (`body`/`response`/`events`), HTTP call body (headers setup, `httpClient.get/post/delete` call) |
| **Python** | `to_str()`, `to_json()`, `from_json()`, `to_dict()`, `from_dict()`, `__properties` ClassVar, `model_config = ConfigDict(...)`, `_with_http_info` method variants, `_without_preload_content` variants, `_serialize` private methods, all `:param _request_timeout:` / `:param _request_auth:` internal parameter docs |
| **C#** | `[DataMember(...)]` attributes, `[JsonConstructorAttribute]` protected default constructor, `ToString()`, `ToJson()`, `IValidatableObject.Validate()` (yield break stub), constructor wiring bodies, `Dispose()`, `GetBasePath()`, private field declarations, `IXxxApiSync`/`IXxxApiAsync` interface declarations (duplicated in concrete class), HTTP implementation bodies (`RequestOptions`, header setup, `Client.Get/Post/Delete` call) |

### Remove for API/service files specifically
- Constructor boilerplate (configuration setup, base path wiring, codec registration) — identical across all API class instances.
- For methods that have three variants (plain / with HTTP info / without preload), keep only the primary variant. The `WithHttpInfo` and `WithoutPreloadContent` variants add no semantic information for training.
- Method bodies that consist only of HTTP plumbing (serialising parameters, setting Accept/Content-Type headers, calling the HTTP client) — replace with `{ /* HTTP implementation */ }` or `...` to preserve the method signature while eliminating noise.

---

## What to keep

### Models
- Class/interface/record declaration including inheritance and interface implementations.
- Class-level XML/Javadoc/docstring summary (describes what the model represents).
- All field/property declarations with their names, types, and nullability.
- All field-level descriptions (the semantic meaning of each field — often the most valuable training signal).
- JSON alias names where they differ from the code name (e.g. `alias="nodeColor"`, `[DataMember(Name="nodeColor")]`).
- Discriminator fields and their default values (e.g. `typeName`, `var_type`).
- Enum value declarations.
- Constructor parameter documentation (parameter names and whether required).
- Polymorphism annotations (`@JsonTypeInfo`, `@JsonSubTypes`, `@JsonIgnoreProperties`, `[JsonDerivedType]`).

### API / service classes
- Class declaration and interface inheritance.
- Every public method's XML/Javadoc/docstring summary and description (explains what the endpoint does).
- Every public method's signature: name, parameters with types, return type.
- `__init__` / constructor for Python/TypeScript (shows the `api_client` dependency injection pattern).

---

## Output format

Separate each file with a level-2 heading and a horizontal rule:

```
## ClassName

<condensed content>

---
```

Collapse three or more consecutive blank lines to one throughout. Remove leading/trailing whitespace per section.

---

## Verification checklist

Before finalising, verify:

- [ ] No license/API-description block appears more than once (only in the header).
- [ ] No import statements remain in model sections.
- [ ] At least one field description is visible in a spot-checked model.
- [ ] At least one method description is visible in a spot-checked API class.
- [ ] Enum values are present in at least one spot-checked enum.
- [ ] No HTTP plumbing code remains (`RequestOptions`, `httpClient.get`, `call_api`, `Client.Get`).
- [ ] File size reduction is at least 70% for model-heavy files.
